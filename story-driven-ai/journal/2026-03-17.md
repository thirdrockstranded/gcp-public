## Observations

### Acceptance Criteria vs. Developer Constitution

During review of US-001, a useful distinction emerged between two types of specification information that both need to exist but belong in different places:

**Acceptance criteria** — what the feature does from the user's perspective. Behavioral. Testable. Story-scoped.

**Developer constitution** — how the developer wants the project to operate, regardless of feature. Cross-cutting. Architectural. Project-scoped.

The "containers by default" decision is a clear example of the second type. It's not specific to photo upload — it applies to every story, every test, every environment. Putting it inside US-001's acceptance criteria would be wrong (it doesn't belong to that story). Leaving it undocumented means an AI generating code will make its own assumptions about environment setup.

**Resolution:** Developer constitution items belong in VISION.md (as guiding constraints/principles) and in CLAUDE.md (so Claude Code reads them at session start). Stories reference the environment implicitly via a standard "Environment / Testing Prerequisites" section that states the Docker expectation explicitly per story, linking back to the project-level default.

---

### The "Testing Environment" Gap in Story Templates

Stories that include E2E or integration tests carry an implicit assumption about what's running when those tests execute. Without stating that assumption, an AI generating test scaffolding may produce:
- Tests that assume a local dev server
- Tests that mock the backend entirely
- Tests that require global installs (Node, Python, browsers) outside of any container

Adding an **Environment / Testing Prerequisites** section to the story template surfaces this assumption explicitly. US-001 is the first story to include this section. It should be backfilled to all stories with E2E or integration test coverage.

**Template addition:** `## Environment / Testing Prerequisites` — standard section, required for any story referencing E2E or integration tests.

---

### Definition of Done vs. Acceptance Criteria

Also surfaced during US-001 review: acceptance criteria tend to be behavioral ("user can do X"), but they don't always capture what "done" means for the implementation. A separate **Definition of Done** section handles:
- Client-side vs. server-side enforcement
- Error message quality standards (no raw exception text)
- Layout/rendering expectations

Added to US-001. Should be added to the story template as an optional-but-recommended section.

---

### Low-Quality Image Warning — Stakeholder Input

Stakeholder (domain expert quilter) confirmed: photo quality does affect pattern quality. A low-resolution source image produces a blurry, pixelated grid that's harder to work from. A non-blocking warning is the right UX — inform but don't block.

Resolution threshold set at 300×300px as a tunable constant. Not a magic number.

---

### File Size Decision

10MB maximum. Rationale: covers virtually all phone camera photos without allowing oversized raw files. Since the image is heavily downsampled during processing anyway, original resolution above a reasonable threshold has no impact on output quality.

## [META] Framework Notes

The distinction between acceptance criteria and developer constitution feels important enough to formalize in the story-driven-ai framework. The current template has no explicit place for project-level technical preferences that affect code generation. CLAUDE.md partially fills this role for Claude Code sessions, but the framework should document the pattern:

- **Story level:** what this feature does + how to verify it + what environment it runs in
- **Project level:** how this project operates (VISION.md constraints + CLAUDE.md)
- **Session level:** what Claude Code knows before touching a story (CLAUDE.md read on session start)

The three-level hierarchy is the answer to "where does this go?" for any piece of information.
