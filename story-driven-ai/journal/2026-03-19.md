## Key Decisions

**Containers by default.** All development, testing, and runtime in Docker. Non-containerized setup is not a supported path. This resolved the "what environment do tests run in?" ambiguity that was flagged as a story template gap.

**US-000 nomenclature.** Pre-product infrastructure bootstrap gets its own story numbered US-000. This keeps the story-as-spec discipline intact while acknowledging that environment setup is a real first step.

**Simplified PR workflow (for now).** Claude Code creates the branch and commits. Developer opens and reviews the PR manually. Automating PR creation is deferred to a later iteration — flagged as a future enhancement to the workflow.

**Stack locked:**
- Frontend: React, TypeScript, Vite, ESLint (TS recommended), Prettier
- Backend: Python, FastAPI, Black, Ruff, type hints required
- Testing: pytest (backend), Playwright (E2E), both containerized
- No local installs assumed outside Docker

**GitFlow conventions.** All work on branches. Branch naming: `feature/US-XXX-short-description`. Never commit to `main` or `develop` directly.

**Session companion documents.** Each story run produces a `US-XXX-name-session.md` alongside the story file. This is the report-back artifact — assumptions (bolded), test results, deviations, methodology flags.

**Story files are read-only for Code.** If a story lacks enough spec to implement correctly, that is a methodology finding, not a reason to edit the story. It goes in the session document flagged `[META]`.

## Observations

### The "Developer Constitution" Pattern

A useful distinction solidified: acceptance criteria describe what a feature does; the developer constitution describes how the project operates. These are different layers and belong in different places. Conflating them produces stories that are either too long or miss project-level constraints entirely.

The three-level hierarchy (story / project / session) is now a named concept in the framework:
- **Story level:** what this feature does, how to verify it, what environment it runs in
- **Project level:** how this project operates (VISION.md + CLAUDE.md)
- **Session level:** what the AI knows before touching a story (CLAUDE.md read at session start)

"Where does this go?" is now answerable by placing the information at the correct level.

### Tooling Choice Delegation

Several stack decisions were made by preference delegation rather than expertise ("I prefer opinionated defaults, you pick"). This is worth noting as a methodology pattern and a risk:

- Black, Ruff, ESLint TS recommended, Vite — all chosen on "most common / most opinionated / simplest" heuristic
- Developer has limited Python depth and delegated those choices explicitly
- This will happen on most real projects. The methodology should address how to handle technical decisions the developer cannot evaluate independently.

This is the specific gap flagged earlier: **what happens when the developer can't evaluate the AI's technical choices in a given language?** It will occur. It occurred here, on the first project.

### Claude Code vs. Cowork

Discussed at session open. Conclusion: Claude Code in VS Code is the right primary tool given developer comfort with terminals and git. Cowork is useful for non-code sessions (doc updates, story refinement) without opening a terminal. The two-tool split maps cleanly onto the PM/implementation project split from the earlier two-Claude-Project methodology.

### CLAUDE.md as Reference Implementation

CLAUDE.md is a Claude Code artifact, not methodology. The concept it embodies — a project briefing document the AI reads at session start — is methodology. The specific file name, format, and behavior are implementation. This distinction is now documented in `framework/scope-and-boundaries.md`.

## [META] Framework Observations

**The template is missing two sections that this session proved necessary:**
1. `## Definition of Done` — behavioral ACs don't fully specify implementation quality. A separate section handles enforcement location, error message standards, rendering expectations, etc.
2. `## Environment / Testing Prerequisites` — required for any story referencing E2E or integration tests. Without it, an AI generating tests makes its own assumptions about what's running.

Both should be added to the story template as standard sections.

**US-000 as a pattern.** Any project using this methodology will have a bootstrap phase that precedes product stories. The US-000 convention is worth formalizing — infrastructure stories that establish the environment are a distinct story type with their own characteristics (no user-facing value, foundational dependency for all other stories, especially high methodology observation value on first run).

**The "extensive" ambiguity.** Using natural language to describe thresholds in workflow instructions ("fix collateral failures if not extensive") is a known failure mode. Any instruction to an AI that relies on judgment about degree needs a concrete number or definition. Resolved here as "more than 2 pre-existing failing tests = flag and stop." The story template and CLAUDE.md pattern should prefer concrete thresholds over qualitative descriptors wherever possible.

**Sycophancy as a methodology risk.** Named explicitly in `scope-and-boundaries.md`. An AI helping build a methodology for working with AI has a structural incentive to validate rather than challenge. The next planned session is a critical review of the methodology — deliberately seeking disconfirmation. This should probably be a recurring practice, not a one-time event. Consider formalizing a "red team" step in the methodology: periodically ask an AI to find what's wrong with the approach, not confirm what's right.

**Separation of implementation from methodology.** The story-driven-ai repo should maintain a clear distinction between what is being researched (the methodology) and what is being used to do the research (Claude, Claude Code, specific workflow patterns). Implementation details should be documented as reference implementations, not presented as universal conclusions. `scope-and-boundaries.md` establishes this. It needs to be actively maintained as the project evolves.
